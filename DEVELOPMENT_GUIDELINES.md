# TourPal Development & AI Assistant Guidelines

## 1. Project Overview
TourPal is a dual-sided management system for inbound tourism. It features an English-language interface for tourists and a Chinese-language interface for local guides. The core tech stack relies on Next.js and MongoDB. 

## 2. AI Assistant (Claude) Core Directives
As an AI coding assistant working on this repository, you **MUST** strictly adhere to the following rules during every session:

### 2.1. The "Changelog-First" Protocol
Whenever you successfully complete a task that involves creating a new feature (`feat`) or fixing a bug (`fix`), you are required to update the `changelog.md` file located in the root directory **before** finishing the conversation or making a final commit.
* **Do not ask for permission** to update the changelog. Do it automatically.
* Categorize the update correctly under `### ✨ Features & Enhancements` or `### 🐛 Bug Fixes`.
* Keep descriptions concise, professional, and focused on business value.

### 2.2. Context Awareness
* Always verify existing types and schemas (especially Prisma/MongoDB schemas) before proposing data structure changes.
* Ensure routing and UI components respect the dual-sided nature of the app (Tourist vs. Guide roles).

## 3. Commit Message Conventions
All commits (whether generated by AI or human developers) must follow the Conventional Commits standard to ensure automated tracking and clear history:

* `feat:` A new feature (e.g., `feat: add guide performance DRE metrics`)
* `fix:` A bug fix (e.g., `fix: resolve checkInTime type conflict`)
* `docs:` Documentation only changes (e.g., `docs: update README with new architecture`)
* `style:` Changes that do not affect the meaning of the code (white-space, formatting)
* `refactor:` A code change that neither fixes a bug nor adds a feature
* `chore:` Changes to the build process or auxiliary tools/libraries

## 4. Team Collaboration Rules
* **Feature Branches:** All new development must occur on separate branches, not directly on `main`.
* **Pull Requests:** When submitting a PR, ensure the PR title matches the commit message conventions above. 
* **Environment Validation:** Before pushing, ensure the Next.js build passes and MongoDB proxy connections (if applicable) are correctly configured.
